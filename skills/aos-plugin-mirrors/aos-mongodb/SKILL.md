# AOS MongoDB (v3.0)

**Core Alignment**: Evidence-first, OPSEC-hardened, fully replicable framework with strong governance and auditability standards.

## Purpose
AOS MongoDB is the AOS mirror of the Grok Build MongoDB plugin. It enables data exploration, collection management, query optimization patterns, and evidence-gated database workflows using local tools.

It is triggered on MongoDB data exploration, Grok Build MongoDB mirror, AOS database skills, or collection management in AOS contexts.

v3.0 adds mandatory Layer-0 on Tier 1 database or data exploration work and strengthened governance integration.

## v3.0 Alignment (Mandatory)
- **Layer-0 Pre-Filter**: Non-bypassable on all Tier 1 database, data exploration, or collection management work. Universe Truth Gate 0 + evidence-gate tri-state + shatter-protocol must inform all query and data decisions.
- **4-Agent Orchestration Default**: Often acts as a specialized database/data agent within 4-Agent workflows for MongoDB or data exploration tasks.
- **Working Document Persistence**: All data exploration sessions, query results, collection management decisions, and tri-state assessments **must** be logged to working-doc-manager.
- **Tri-State Evidence (+1 / 0 / -1)**: All major data findings and query conclusions carry explicit tri-state.
- **Governance Enforcement**: Any data exploration or query that bypasses evidence standards or introduces unverified claims triggers revision.

## Integration Points
- **Layer-0** — Enforces pre-filter passage on all Tier 1 database or data exploration work.
- **build-coordinator** — Primary collaborator on database or data pipeline builds.
- **working-doc-manager** — All data exploration sessions and query results are logged here.
- **reasoning-architect** — Often collaborates on reasoning aspects of data analysis and query logic.

## OPSEC & Frog Protocol (Mandatory)
- **Frog Protocol**: Sensitive collections, queries, or data patterns receive stable codenames or abstraction levels.
- **Private by Default**: Detailed data exploration reasoning and internal query analysis remain in the working document. Only final findings and high-level recommendations are surfaced unless explicit governance/audit mode is requested.
- **Evidence-First, No Self-Report Reliance**: Never accepts self-reported data quality or query results. All findings are grounded in actual exploration and logs.
- **Replicability & Auditability**: Every data exploration session produces a clear, logged analysis suitable for later review.

## Best Practices
- Always enforce Layer-0 on Tier 1 database or data exploration work.
- Maintain clear evidence trails from queries and data exploration.
- Log all sessions and tri-state assessments.
- Combine with reasoning-architect for complex data analysis.
- Keep data conclusions strictly evidence-based.
