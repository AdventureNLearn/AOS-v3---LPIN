# AOS Vercel (v3.0)

**Core Alignment**: Evidence-first, OPSEC-hardened, fully replicable framework with strong governance and auditability standards.

## Purpose
AOS Vercel is the AOS mirror of the Grok Build Vercel plugin. It provides deployment management, build status checking, domain configuration patterns, and evidence-gated deployment workflows.

It is triggered on Vercel deployment, Grok Build Vercel mirror, AOS deploy skills, or manage Vercel projects in AOS contexts.

v3.0 adds mandatory Layer-0 on Tier 1 deployment or hosting work and strengthened governance integration.

## v3.0 Alignment (Mandatory)
- **Layer-0 Pre-Filter**: Non-bypassable on all Tier 1 deployment, hosting, or Vercel project management work. Universe Truth Gate 0 + evidence-gate tri-state + shatter-protocol must inform all deployment and configuration decisions.
- **4-Agent Orchestration Default**: Often acts as a specialized deployment/hosting agent within 4-Agent workflows for Vercel or deployment tasks.
- **Working Document Persistence**: All deployment decisions, build status checks, domain configurations, and tri-state assessments **must** be logged to working-doc-manager.
- **Tri-State Evidence (+1 / 0 / -1)**: All major deployment and configuration recommendations carry explicit tri-state.
- **Governance Enforcement**: Any deployment or configuration that bypasses evidence standards or introduces unvetted changes triggers revision.

## Activation Triggers
**Explicit**: "Vercel deployment", "Grok Build Vercel mirror", "AOS deploy skills", "manage Vercel projects in AOS"
**Implicit**: Any work involving Vercel deployment, hosting, or project management within AOS contexts.

## Integration Points
- **Layer-0** — Enforces pre-filter passage on all Tier 1 deployment or hosting work.
- **build-coordinator** — Primary collaborator on deployment and hosting builds.
- **working-doc-manager** — All deployment decisions and build status checks are logged here.
- **reasoning-architect** — Often collaborates on reasoning aspects of deployment architecture and status analysis.

## OPSEC & Frog Protocol (Mandatory)
- **Frog Protocol**: Sensitive deployments, domains, or projects receive stable codenames or abstraction levels.
- **Private by Default**: Detailed deployment reasoning and internal configuration iterations remain in the working document. Only final deployment status and high-level recommendations are surfaced unless explicit governance/audit mode is requested.
- **Evidence-First, No Self-Report Reliance**: Never accepts self-reported deployment success or configuration validity. All decisions are logged and verified.
- **Replicability & Auditability**: Every deployment produces a clear, logged history suitable for later review.

## Best Practices
- Always enforce Layer-0 on Tier 1 deployment or hosting work.
- Maintain clear evidence trails from build status and deployment logs.
- Log all deployment decisions and tri-state assessments.
- Combine with reasoning-architect for complex deployment architecture decisions.
- Keep deployment conclusions strictly evidence-based.
