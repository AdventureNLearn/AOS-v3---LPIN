# Grok Build Plugins (v3.0)

**Core Alignment**: Evidence-first, OPSEC-hardened, fully replicable framework with strong governance and auditability standards.

## Purpose
Grok Build Plugins is used for Grok Build Plugin Marketplace queries, official plugin listings, AOS compatibility notes, or scaffolding local equivalents for dev tool integrations (MongoDB, Vercel, Sentry, Chrome DevTools, Cloudflare, Superpowers). It is triggered on phrases like "Grok plugins", "Grok Build marketplace", "make plugins available here", or "Grok Build skills".

v3.0 adds mandatory Layer-0 on Tier 1 plugin or marketplace work and strengthened governance integration.

## v3.0 Alignment (Mandatory)
- **Layer-0 Pre-Filter**: Non-bypassable on all Tier 1 plugin marketplace, compatibility, or scaffolding work. Universe Truth Gate 0 + evidence-gate tri-state + shatter-protocol must inform all queries and recommendations.
- **4-Agent Orchestration Default**: Often acts as a specialized plugin/marketplace agent within 4-Agent workflows for plugin or dev tool integration tasks.
- **Working Document Persistence**: All query results, compatibility notes, scaffolding decisions, and tri-state assessments **must** be logged to working-doc-manager.
- **Tri-State Evidence (+1 / 0 / -1)**: All major plugin recommendations and compatibility assessments carry explicit tri-state.
- **Governance Enforcement**: Any recommendation that bypasses evidence standards or introduces unvetted plugins triggers revision.

## Core Capabilities

**Marketplace & Plugin Queries**  
Handles queries about official Grok Build plugins, listings, and availability.

**Compatibility Analysis**  
Provides AOS compatibility notes for official plugins and dev tool integrations.

**Local Equivalent Scaffolding**  
Designs and scaffolds local/open-source equivalents for dev tool integrations (MongoDB, Vercel, Sentry, Chrome DevTools, Cloudflare, Superpowers) when full self-sovereignty or customization is required.

**Integration Guidance**  
Offers implementation guidance for integrating plugins or their local equivalents into AOS workflows.

## Activation Triggers
**Explicit**: "Grok plugins", "Grok Build marketplace", "make plugins available here", "Grok Build skills"
**Implicit**: Any work involving plugin discovery, compatibility assessment, or scaffolding local equivalents for dev tool integrations.

## Integration Points
- **Layer-0** — Enforces pre-filter passage on all Tier 1 plugin or marketplace work.
- **build-coordinator** — Primary collaborator when integrating plugins into builds.
- **working-doc-manager** — All query results, compatibility notes, and scaffolding decisions are logged here.
- **aos-*-* plugin mirrors** (e.g., aos-mongodb, aos-vercel) — Provides the canonical local equivalents and integration patterns.

## OPSEC & Frog Protocol (Mandatory)
- **Frog Protocol**: Sensitive plugin usage, custom integrations, or scaffolding logic receive stable codenames when operational sensitivity exists.
- **Private by Default**: Detailed compatibility reasoning and internal scaffolding iterations remain in the working document. Only final recommendations and high-level guidance are surfaced unless explicit governance/audit mode is requested.
- **Evidence-First, No Self-Report Reliance**: Never accepts self-reported plugin quality or compatibility. All recommendations are grounded in verified information and logged.
- **Replicability & Auditability**: Every plugin recommendation or scaffolding decision produces a clear, logged rationale suitable for later review.

## Best Practices
- Always enforce Layer-0 on Tier 1 plugin or marketplace work.
- Clearly distinguish between official plugins and local/self-sovereign equivalents.
- Log all compatibility assessments and scaffolding decisions with tri-state.
- Prioritize evidence-backed recommendations over convenience.
- Maintain awareness of governance implications when recommending plugin integrations.
