# Solo Operator Playbook (v3.0)

**Core Alignment**: Evidence-first, OPSEC-hardened, fully replicable framework with strong governance and auditability standards.

## Purpose
Solo Operator Playbook is a persistent reference for solo operators focused on zero-cost bandwidth maximization and flexible drift protection. It provides activation triggers, integration patterns with core verification and governance skills, and a living reference for maintaining high performance in independent work.

v3.0 updates it for full alignment with Layer-0, 4-Agent patterns (even in solo mode), and strengthened governance integration.

## v3.0 Alignment (Mandatory)
- **Layer-0 Pre-Filter**: Non-bypassable on all high-stakes solo work. The playbook explicitly requires running Universe Truth Gate 0 + evidence-gate + shatter-protocol before major decisions or deliverables.
- **4-Agent Orchestration Default** (Solo Adaptation): Even in solo mode, the operator simulates Agent roles internally (Coordinator, Reasoning, Verification, Execution) with explicit handoffs logged to working document.
- **Working Document Persistence**: All major decisions, bandwidth allocation, drift checks, and tri-state results **must** be logged to working-doc-manager.
- **Tri-State Evidence (+1 / 0 / -1)**: All major solo decisions and outputs carry explicit tri-state.
- **Governance Enforcement**: Any detected drift, anti-pattern, or governance weakening in solo mode triggers immediate self-correction or escalation to mission-spine-guard.

## Core Principles

**Zero-Cost Bandwidth Maximization**  
Focus on high-leverage activities that deliver maximum value with minimal overhead.

**Flexible Drift Protection**  
Maintain mechanisms to detect and correct drift in reasoning, scope, or governance without heavy process overhead.

**Internal 4-Agent Simulation**  
Even when working alone, consciously rotate through Coordinator, Reasoning, Verification, and Execution roles with clear handoffs logged in the working document.

**Evidence-First Solo Mode**  
Never bypass Layer-0 gates, even in solo work. Evidence quality and governance remain non-negotiable.

## Activation Triggers
**Explicit**: "solo operator playbook", "maximize bandwidth", "harden drift protection", "solo operator mode"
**Implicit**: Any period of independent, high-stakes, or long-horizon work where the operator needs structured self-governance and drift protection.

## Integration Points
- **evidence-gate** & **shatter-protocol** — Mandatory Layer-0 gates for solo high-stakes work.
- **anti-pattern-scanner** — Regular self-audit for common solo operator failure modes.
- **ops-hardening-architect** — Provides hardening patterns for solo workflows.
- **mission-spine-guard** — Escalation target when solo drift or governance issues are detected.
- **working-doc-manager** — Primary state, decision, and audit logging mechanism.

## OPSEC & Frog Protocol (Mandatory)
- **Frog Protocol**: Sensitive projects or investigations receive stable codenames even in solo work.
- **Private by Default**: Detailed internal reasoning and self-audits remain in the working document. Only final outputs are surfaced unless explicit governance/audit mode is requested.
- **Evidence-First, No Self-Report Reliance**: Never accepts self-reported quality or alignment in solo mode. All decisions are logged and verified through Layer-0 gates.
- **Replicability & Auditability**: Every solo run produces a clear, logged history suitable for later self-review or handoff.

## Best Practices
- Always run Layer-0 gates before major solo decisions or deliverables.
- Maintain a living working document with explicit role handoffs (even when simulated internally).
- Regularly run anti-pattern-scanner on your own workflow.
- Use the working document as the single source of truth for bandwidth allocation and drift checks.
- Escalate to mission-spine-guard (or external review) when persistent drift or governance issues appear.
